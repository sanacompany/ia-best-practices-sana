¿En qué nivel estás?

Niveles 1–4

Del cero al autocomplete

1 Sin IA

Cero herramientas de IA. Escribiendo todo a mano.

2 Agente en IDE + Permisos

Cursor/Copilot integrado. Permisos activados, revisás todo.

3 Modo YOLO

Confianza creciente. Menos restricciones, dejás que el agente haga más.

4 Menos Code Review

Ya no mirás cada diff. Te enfocás en qué hace el agente, no en el código línea por línea.

La mayoría de equipos están entre nivel 2 y 3. El salto a nivel 4 requiere buenas rules y contexto.
Niveles 5–8

Del agente único al orquestador

5 Agent-First

El agente lidera. Vos mirás el código después, no durante.

6 Múltiples Agentes

Varios agentes en paralelo. Multiplexing entre terminales y chats.

7 Orquestación Manual

10+ agentes. Coordinación a mano = caos. “Le mandé el mensaje al agente equivocado”.

8 Orquestador Custom

Construiste tu propia infraestructura para coordinar agentes.

Meta realista para SANA: Llegar a nivel 5–6 con las herramientas correctas, rules sólidas, y Claude Code + Cursor bien configurados.

Cursor

Fundamentos que cambian todo

  • Contexto es rey. Cursor solo es tan bueno como el contexto que le das.
  • @file, @folder, @web — Referenciá archivos específicos en el chat siempre.
  • Composer > Chat para cambios multi-archivo. Chat para preguntas.
  • Cmd+K para edición inline rápida en el archivo actual.
  • Codebase indexing — Asegurense que esté activado en Settings.
  • .cursorignore — Excluyan node_modules, builds, y archivos grandes.
  • Modelo correcto — Claude Sonnet 4 para speed, Opus para complejidad.
  • Atajos clave: Cmd+L chat, Cmd+I composer, Cmd+K inline.
Cursor Rules

Cómo crear sus propias rules

Las rules le dicen a Cursor cómo debe trabajar en su proyecto.

# .cursor/rules/api-standards.mdc --- description: "REST API guidelines" globs: "src/api/**/*.ts" alwaysApply: false --- # API Rules - Usar JSON response: { data, error, meta } - Status codes correctos (201, 204, etc) - Validar input con zod - Loguear errores con request context

Estructura recomendada

.cursor/ rules/ base.mdc # Siempre activa react.mdc # Para .tsx api.mdc # Para src/api/ testing.mdc # Para *.test.*
Tip: /create-rule en Cursor genera rules automáticamente.

3 modos: Always Apply, Apply to Files (globs), Apply Intelligently (description).

Cursor

Best practices para rules

DO
  • Una preocupación por archivo de rule
  • Menos de 500 líneas por rule
  • Instrucciones concretas y verificables
  • Globs específicos, no genéricos
  • Referenciar archivos del proyecto real
DON'T
  • Duplicar lo que hace el linter
  • Copiar código entero en la rule
  • Rules vagas como “escribi buen código”
  • Un solo archivo gigante para todo
  • Olvidarse de revisar si siguen siendo válidas
Acción inmediata: Cada proyecto de SANA debería tener .cursor/rules/base.mdc con el stack, convenciones y comandos principales.

El CLI que piensa como un dev senior

Claude Code

CLAUDE.md — Tu copiloto configurado

El equivalente a Cursor Rules pero para Claude Code. Se carga automáticamente en cada sesión.

# CLAUDE.md ## Stack Next.js 15, TypeScript, Prisma, tRPC ## Comandos - npm run dev — servidor local - npm test — correr tests - npm run lint — lint + format ## Convenciones - 2-space indent - Functional components con hooks - API handlers en src/api/handlers/

Dónde viven

# Proyecto (compartido via git) ./CLAUDE.md ./.claude/CLAUDE.md # Personal (todos tus proyectos) ~/.claude/CLAUDE.md # Local (solo vos, no git) ./CLAUDE.local.md # Rules modulares .claude/rules/ code-style.md testing.md security.md

Tip: /init genera un CLAUDE.md analizando tu codebase.

Claude Code

Rules con path-scoping

Igual que las globs de Cursor, podés tener rules que solo aplican a ciertos archivos.

# .claude/rules/api-rules.md --- paths: - "src/api/**/*.ts" --- # API Development Rules - Validar input con zod - Usar formato estándar de error - Documentar con OpenAPI comments - Loguear errores con request ID

Skills

Workflows reutilizables como comandos

Skills

Extendé Claude con Skills custom

Un Skill = un SKILL.md con instrucciones que Claude ejecuta automáticamente o via /nombre.

# .claude/skills/deploy/SKILL.md --- name: deploy description: Deploy to staging/prod allowed-tools: Bash, Read --- ## Steps 1. Run npm test 2. Run npm run build 3. Ask which environment (staging/prod) 4. Execute deploy script 5. Verify health check

Skills que vienen built-in

  • /simplify — Review de calidad y eficiencia
  • /batch — Cambios masivos en paralelo
  • /debug — Troubleshoot la sesión actual
  • /loop — Ejecutar algo en schedule

Dónde crearlas

# Personal (todos los proyectos) ~/.claude/skills/mi-skill/SKILL.md # Proyecto (compartido con el equipo) .claude/skills/mi-skill/SKILL.md

MCP
Model Context Protocol

Conectá Claude al mundo exterior

MCP

¿Qué es MCP?

Un estándar abierto que conecta a Claude con herramientas externas: bases de datos, APIs, issue trackers, diseño, y más.

Servidores populares

  • GitHub — PRs, issues, code search
  • Notion — Docs, bases de datos
  • Sentry — Errores en producción
  • PostgreSQL — Queries directos
  • Linear — Issues y projects
  • Figma — Diseños a código

Cómo agregar un server

# HTTP (remoto, recomendado) claude mcp add --transport http \ github https://api.githubcopilot.com/mcp/ # Stdio (local) claude mcp add --transport stdio \ --env DB_URL=postgresql://... \ db -- npx -y @bytebase/dbhub # Ver estado /mcp # dentro de Claude Code
Scopes: local (default, personal), project (.mcp.json, compartido), user (global).
MCP

MCP en la práctica para SANA

Proyecto compartido (.mcp.json)
{ "mcpServers": { "github": { "type": "http", "url": "https://api.githubcopilot.com/mcp/" }, "sentry": { "type": "http", "url": "https://mcp.sentry.dev/mcp" } } }
Casos de uso
  • “Mirá los errores de Sentry de las últimas 24h y proponé fixes”
  • “Creá un PR con los cambios que hicimos”
  • “Trae el diseño de Figma y implementá el componente”
  • “Query la tabla de usuarios con más de 30 días sin login”
  • “Actualizá el doc en Notion con el nuevo endpoint”

MCP también funciona en Cursor. Es un estándar compartido.

SubAgents

Trabajo en paralelo con agentes especializados

SubAgents

Agentes especializados a tu servicio

Explore

Modelo rápido (Haiku). Solo lee archivos. Ideal para descubrir código y buscar en el codebase.

Plan

Solo lectura. Investiga el codebase durante el modo plan para diseñar una estrategia.

General-Purpose

Acceso a todas las herramientas. Para tareas complejas multi-paso.

Key insight: Los subagents corren en su propio contexto → no contaminan tu conversación principal. Podés lanzar varios en paralelo.
SubAgents

Creá tus propios agentes

# .claude/agents/code-reviewer.md --- name: code-reviewer description: Expert code review tools: Read, Glob, Grep, Bash model: sonnet --- You are a senior code reviewer. When invoked: 1. Run git diff to see changes 2. Focus on modified files 3. Check for: - Code clarity - Duplicated code - Error handling - Exposed secrets - Test coverage Output: Critical > Warnings > Suggestions

Configuración avanzada

  • model: haiku — barato y rápido para tareas simples
  • model: opus — máxima capacidad para tareas complejas
  • isolation: worktree — git worktree aislado
  • background: true — siempre en background
  • memory: project — persiste conocimiento entre sesiones
  • maxTurns: 10 — limitar rondas

Dónde guardarlos

~/.claude/agents/ # Personal .claude/agents/ # Proyecto

Tip: /agents para ver y crear agentes.

Workflows

El poder del trabajo en paralelo

Esto es lo que separa nivel 3 de nivel 6. Aprendé a delegar.

Antes (secuencial)
  1. Investigar el bug
  2. Entender el código
  3. Escribir el fix
  4. Escribir tests
  5. Hacer el PR

Todo en una sola conversación

Ahora (paralelo)

Agent 1: Investiga y planea el fix

Agent 2: Revisa tests existentes

Agent 3: Busca patrones similares en el repo

Tú: Revisás los resultados, implementas con contexto completo

Ctrl+B para mandar al background

Pro Tips

Tips que hacen la diferencia

Prompt Engineering Básico
  • Se específico: “usá zod para validar” > “validá el input”
  • Dá contexto: “en este archivo, la función X hace Y”
  • Un paso a la vez para cambios grandes
  • Pegá screenshots de errores, no los describas
Workflow Tips
  • Plan mode primero, implementar después
  • /compact cuando la conversación se hace larga
  • Commits pequeños y frecuentes
  • Nunca confiar ciegamente — revisar el output
Claude Code Específicos
  • claude --resume para continuar sesiones
  • /memory para auto-memoria persistente
  • /init para bootstrap de proyecto nuevo
  • Multimodal: pegar imágenes directo en el CLI
Cursor Específicos
  • Usar @codebase para búsquedas amplias
  • Composer para multi-archivo, Chat para preguntas
  • AGENTS.md como alternativa simple a rules
  • Verificar que codebase indexing esté ON
